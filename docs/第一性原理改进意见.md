# taotao-skills 第一性原理改进意见

> 分析日期：2026-07-21 | 分析范围：ARCHITECTURE.md、README.md、技能体系蓝图.md、12个技能的SKILL.md
> 分析视角：回归"教师到底在做什么"，而非"我有哪些工具"

---

## 一、现有体系的结构性成就（先肯定）

在提改进意见之前，必须承认现有设计已经具备**优秀架构的骨架**：

1. **三层架构（L0编排/L1能力/L2数据）**是正确的方向，与教师工作流天然对齐
2. **单一数据源原则**（guwen-cidian为中枢）避免了数据碎片化
3. **frontmatter约定**（category/depends-on/display-name）为机器可读化打下基础
4. **用户分层矩阵**（新手/毕业班/文言专长/教研命题）体现了以用户为中心
5. **护栏内建理念**（doc-review收口）是质量意识的前置化
6. **"定评vs商榷"机制**（gushi-appreciation）突破了AI输出"唯一答案"的局限

这些问题如果放在其他项目里可能根本不存在。**以下改进意见针对的是"从好到更好"，不是"从错到对"。**

---

## 二、第一性原理诊断：从"教师工作流"出发的五个根本缺口

### 缺口1：工作流闭环断裂——"教"与"练/批"几乎为零

**第一性原理**：教师的时间分配大约是——备课20%、课堂实施40%、作业批改与反馈20%、评价与教研20%。

**现状对照**：

| 工作流环节 | 时间占比 | 现有技能覆盖 | 覆盖率 |
|---|---|---|---|
| 备（备课设计） | ~20% | 备课助手、整本书阅读、跨学科设计、配图规划、古诗文鉴赏 | ✅ 高 |
| 教（课堂实施） | ~40% | **零** | ❌ **零** |
| 练/批（作业反馈） | ~20% | wordfreq（文本分析，边缘）、作文批改（规划中） | ⚠️ 极低 |
| 评（考情研判） | ~10% | 江西中考备考、中考分析 | ✅ 高 |
| 研（教研写作） | ~10% | 教研写作、文稿审核、王力/吕叔湘视角 | ✅ 高 |
| 输出/传播 | ~5% | 配图规划 | ⚠️ 低 |
| 成长（专业学习） | ~5% | 王力/吕叔湘视角（被动顾问型） | ⚠️ 低 |

**核心问题**：当前体系是"备课设计师的豪华工具箱"，不是"教师的完整操作系统"。教师70%的时间（教+练批）几乎没有被覆盖。

**这不是"缺几个技能"的问题，而是架构层面的域失衡。**

---

### 缺口2：L0编排层从未落地——"操作系统"仍是蓝图

**第一性原理**：教师使用AI时，真实的表达方式是——"帮我备一节《岳阳楼记》"，而不是"先调用备课助手，再调用文言词典，再调用文稿审核"。

**现状**：
- `teacher-workflow` 元技能在ARCHITECTURE.md中标记为"规划中"，至今不存在
- README中的"典型工作流"表格（如"从零备一节课"的串链）是**纯文本说明**，不是可执行代码
- 用户分层入口矩阵是静态表格，不是动态路由

**结果**：新手教师面对12个技能，仍然不知道"我说一句话，它该调哪几个"。这违背了"do not make users think"的设计原则。

**关键洞察**：
> 没有编排层的技能体系，无论单个技能多精致，本质上仍是"用户自己当调度器"。这与"AI助手"的承诺相悖——助手应该理解意图并自动编排，而不是让用户记住每个工具的用途。

---

### 缺口3：依赖关系是"君子协定"，不是"机器契约"

**第一性原理**：如果doc-review声明了`depends-on: [guwen-cidian]`，那么当guwen-cidian的词条模板变更时，doc-review的古文引用审核流程应该**自动感知**，而不是靠作者记忆去同步。

**现状**：
- `depends-on` 写在frontmatter里，但只是**声明式元数据**，没有任何运行时语义
- gushi-appreciation的`depends-on: [guwen-cidian, doc-review]`无法被任何工具验证
- guwen-cidian/references/下的文件结构变更（如重命名、移动）会静默破坏下游技能
- doc-review的"古文引用专项审核"中说"数据回查guwen-cidian"，但路径是硬编码在prompt文本中的

**风险示例**：
```
guwen-cidian/references/文言词表/词条标准模板.md
                         ↓ 某天更名为"词条录入规范.md"
doc-review SKILL.md中的prompt仍引用"词条标准模板.md"
                         ↓ 运行时找不到文件，但无报错机制，静默降级
```

---

### 缺口4：质量护栏是"可选配件"，不是"默认配置"

**第一性原理**：好系统的设计原则是"默认正确"。如果doc-review是护栏，那么所有生产类技能的默认行为应该是"产出经审核"，而不是"产出后 optionally 调审核"。

**现状**：
- doc-review是独立技能，需要用户**主动调用**
- 教研写作、备课助手、古诗文鉴赏等生产类技能，默认不串doc-review
- 品牌声音规范（docs/品牌声音与品牌规范.md）是"参考"而非"强制"

**这与"护栏内建"的设计原则相悖。** 当前实现的是"护栏旁置"——护栏存在，但不挡住危险行为。

---

### 缺口5：用户入口是"身份标签"，不是"意图表达"

**第一性原理**：教师打开AI时，心里想的是"我要备一节课""我要改作文""我要分析这次月考"——这些是**意图**，不是"我是新手教师"这种**身份**。

**现状**：
- README的入口是"你是…？先装这几个"
- ARCHITECTURE.md的用户分层矩阵也是按角色分
- 没有"意图→技能链"的映射层

**问题**：一个"中考毕业班教师"也可能今天想写教研论文、明天想备新课。按身份分入口是静态的，按意图分入口才是动态的，才符合真实使用场景。

---

## 三、改进建议：从第一性原理推导的五个行动

### 行动1：补全工作流闭环——新增"课堂实施域"与"作业反馈域"

**不是加技能，是加域。** 现有4个域需要扩展为6个域：

```
原4域：备课设计 | 文言古汉语 | 评价考情 | 教研写作
新6域：备课设计 | 课堂实施 | 作业反馈 | 文言古汉语 | 评价考情 | 教研写作 + 传播
```

**课堂实施域**（高优先级，解决"教"的真空）：
- 课堂提问设计器：输入教学目标，输出层层递进的问题链（参考SOLO分类理论）
- 互动活动生成器：小组讨论/角色扮演/辩论/思维导图活动的任务卡
- 板书与PPT文案设计：不是课件生成，是"屏幕上该出现什么字、什么结构"

**作业反馈域**（高优先级，解决"练/批"的真空）：
- 作文批改助手（已规划中）：PII本地化，只给反馈不判分，避免合规红线
- 作业设计器：分层作业（基础/提高/拓展）+ 答案与评分标准
- 错题归因分析：学生常见错误模式 → 对应补救教学策略

**为什么不是简单加技能，而是加域？** 因为域是架构层面的分类，影响frontmatter的category字段、README的组织方式、用户的心智模型。如果"课堂实施"只是散落在备课设计域里，教师永远找不到。

---

### 行动2：落地L0编排层——用"意图路由"替代"身份入口"

**不要等一个完美的teacher-workflow元技能，先做一个最小可用的"意图-技能链映射表"。**

**具体做法**：

在README和ARCHITECTURE.md之间，新增一个`INTENT_ROUTING.md`：

```markdown
# 意图路由表（Intent → Skill Chain）

| 用户意图 | 触发关键词 | 推荐技能链 | 默认审核 |
|---|---|---|---|
| 备一节新课 | "备一节课"/"教案"/"教学设计" | k12-lesson-planning-zh → guwen-cidian(若含文言) → doc-review | ✅ |
| 备古诗文赏析 | "古诗鉴赏"/"赏析备课" | gushi-appreciation → guwen-cidian → doc-review | ✅ |
| 写教研论文 | "论文"/"课题"/"教研写作" | jiaoyan-writing → wangli/lvshuxiang-perspective(可选) → doc-review | ✅ |
| 分析中考真题 | "中考"/"命题规律"/"考情" | jiangxi-zhongkao-beikao / zhongkao-analysis → wordfreq(文本分析) | ❌ |
| 整本书阅读设计 | "名著"/"整本书"/"导读" | whole-book-reading → article-illustration(配图) | ❌ |
| 审核我写的东西 | "审核"/"校对"/"润色" | doc-review | — |
```

**为什么这个表比"身份入口"更好？**
- 一个教师可以有多重意图，但只有一个身份
- 意图是动词性的（"我要备..."），与技能链的动词-名词命名一致
- 可以作为teacher-workflow元技能的训练数据

**下一步**：将这个表转化为一个真正的`teacher-workflow` SKILL.md——它本身不生产内容，只做"意图识别 → 技能路由 → 结果收口"。

---

### 行动3：将依赖关系从"声明式"升级为"可验证式"

**核心思路**：让`depends-on`不只是frontmatter里的文字，而是可以被脚本验证的契约。

**具体做法**：

A. **定义依赖契约文件**
在每个被依赖技能（如guwen-cidian）的根目录下，新增一个`API.md`（或`CONTRACT.md`），明确承诺：
```markdown
# guwen-cidian 对外契约

## 稳定接口（改名/移动需发breaking change）
- `references/文言词表/江西省中考文言文高频词表.md` — 104词高频表
- `references/文言词表/词条标准模板.md` — 新增词目规范
- `references/古诗文基础/初中文言文通假字.md` — 通假字数据
- `references/教材版本/统编教材版本对照表.md` — 版本核对

## 输出格式契约
- 字词卡片格式（见SKILL.md模式一）
- 串讲卡格式（见SKILL.md模式三）

## 版本兼容性
- v2.x 保证以上接口稳定
- v3.0 可能重构目录结构，届时发migration guide
```

B. **添加依赖验证脚本**
在仓库根目录新增`scripts/verify-deps.py`：
```python
# 遍历所有SKILL.md，读取depends-on
# 检查被依赖技能的API.md中是否存在声明的接口
# 检查被依赖技能的路径引用是否在引用方的prompt中真实存在
# 输出：哪些依赖是健康的，哪些是broken link
```

C. **在贡献指南中强制要求**
任何修改guwen-cidian/references/目录结构的PR，必须同时更新：
1. guwen-cidian/API.md
2. 所有依赖它的技能的SKILL.md中的路径引用
3. verify-deps.py通过

---

### 行动4：将doc-review从"旁置护栏"改为"默认收口"

**核心思路**：不改变doc-review作为独立技能的存在，但让生产类技能**默认调用**它。

**具体做法**：

A. **在SKILL.md中增加`default-review: true`字段**
生产类技能（备课助手、教研写作、古诗文鉴赏）的frontmatter：
```yaml
---
name: jiaoyan-writing
category: 教研写作
default-review: true   # 新增：产出默认经doc-review收口
review-scope: [古文引用, 术语一致性, 语体规范]  # 可选：只审特定关卡
---
```

B. **在teacher-workflow编排层实现默认串链**
当用户通过意图路由触发"写教研论文"时，编排层自动：
```
jiaoyan-writing → doc-review（自动，无需用户显式要求）
```
并提示："产出已自动经文稿审核，如需跳过加 --skip-review"

C. **品牌声音规范从"参考"升级为"校验清单"**
将docs/品牌声音与品牌规范.md转化为一个可checklist：
```markdown
## 品牌声音校验清单（生产类技能产出必过）

- [ ] 语气准确、干净、不浮夸
- [ ] 不堆砌术语（尤其面对学生/家长的内容）
- [ ] 古文引用注明出处与版本
- [ ] 区分"教材定评"与"可讨论点"
- [ ] 不生成"唯一标准答案"让学生背诵
```
doc-review的第四关（文风与教学规范关）直接引用此清单，逐项勾选。

---

### 行动5：重构用户入口——从"你是谁"到"你要做什么"

**具体做法**：

A. **README首屏改为意图导向**
```markdown
## 我想...（按意图选入口）

| 我想... | 推荐技能链 |
|---|---|
| 备一节课 | 备课助手 → （文言/古诗内容）→ 文稿审核 |
| 准备中考复习 | 江西中考备考 / 中考分析 → 整本书阅读 |
| 写教研论文/课题 | 教研写作 → 王力/吕叔湘视角 → 文稿审核 |
| 查文言文/古诗字词 | 文言词典 / 古诗文鉴赏 |
| 分析学生作文/文本 | 文本词频 / （作文批改，规划中） |
| 设计名著阅读课 | 整本书阅读 → 配图规划 |
| 审核我写的内容 | 文稿审核 |
```

B. **保留身份矩阵作为次级导航**
身份矩阵不删，放在"不知道选哪个？先对号入座"的位置，作为辅助而非主入口。

C. **每个SKILL.md的"何时使用"改为"解决什么问题"**
从功能触发式：
> "当用户需要审核/校对/润色...时触发"

改为目标导向式：
> "当你写完教案、论文、推文，需要确保零语病、零错字、古文引用准确无误时"

## 五、一个更深层的追问

以上五个行动解决的是"从好到更好"。但如果真的要回到第一性原理，还有一个更深层的问题：

> **这个技能体系的终极形态，是教师个人的"教学操作系统"，还是一个可共享的"教学技能市场"？**

如果是**个人操作系统**：
- 应该更强调teacher-workflow的个性化（"我的常用链"）
- 应该允许教师自定义frontmatter（"我常备古诗课，把gushi-appreciation置顶"）
- 数据底座可以更深（我的学生错题库、我的常用评语库）

如果是**共享技能市场**：
- 应该更强调标准化（统一的frontmatter、统一的API契约）
- 应该降低贡献门槛（让其他教师也能提交技能）
- 应该强调可组合性（技能像乐高积木）

当前设计在这两者之间有点摇摆：ARCHITECTURE.md像市场规范，但guwen-cidian里的江西中考数据又像个人沉淀。这不是对错问题，但值得你在v2阶段明确战略定位——**它决定了teacher-workflow是"个人助理"还是"公共资源调度器"，也决定了缺口技能的优先级排序。**

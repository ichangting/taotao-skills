## 二、当前体系本质：自发聚成的"四域工具箱"

12 个技能沿真实需求自然长出来，沉淀为 4 个能力域：

- **备课与资源**（4）：备课、整本书、跨学科、配图
- **文言与古汉语**（3）：文言词典（已是数据中枢）+ 两位学者视角（专家顾问型）
- **评价与考情**（3）：江西中考、通用中考分析、词频
- **教研与写作**（2）：教研写作 + 文稿审核（护栏）

它的真实状态是：**好用的工具箱，但不是系统**。技能间怎么串、谁依赖谁，只存在于 README 的几行表格和 prompt 文本里，没有机器可读的契约。

## 三、第一性原理：教师到底在做什么

剥离工具，回归教师价值闭环：

**备 → 教 → 练/批 → 评 → 研 → 输出/传播 → 成长**

- 备：把课标/教材/学情转化为课堂方案（教材解读、教法设计、资源生产）
- 教：课堂实施（提问、互动、板书）
- 练/批：作业、批改、反馈
- 评：命题、考情研判、质量分析
- 研：听课评课、写论文/课题、专业阅读
- 输出/传播：写文章、公众号、配图、审核
- 成长：个人知识管理、读学者原著

同时教师不是单一角色，应分**用户层**：新手教师 / 熟手教师 / 中考毕业班教师 / 文言专长教师 / 教研命题者 / 学生（未来）。

## 四、产品诊断：当前体系的 6 个问题

1. **只有工具箱，没有"操作系统"**：无编排/路由层，新手不知先装哪个；串联规则只在 README 文本里。
2. **命名与定位不统一**：功能名（wordfreq、doc-review）、场景名（whole-book-reading）、人物名（wangli-perspective）三种风格混用，教师看名猜不出能干啥。
3. **覆盖不均且有真空**：教（课堂实施）全缺；学生端为零；作文批改、古诗文鉴赏、课件生成空缺。而"中考分析"有两个技能职责边界模糊。
4. **数据与依赖隐式脆弱**：guwen-cidian 是中枢，doc-review 回查它，但依赖写在 prompt 文本，无 manifest；改名即断。
5. **缺用户分层与场景入口**：12 个技能一视同仁，没有"新手三件套""中考包"等按需组合。
6. **护栏未内建**：doc-review 是写后审核，但生产类技能（教案、论文）默认不串它，质量靠自觉。

## 五、目标体系蓝图：工作流驱动的三层架构

### L0 编排层（Orchestration）
- `teacher-workflow` 元技能：识别教师意图 → 路由到技能链 → 串联数据流 → 产出默认过 doc-review
- 场景包（组合预设）：备课包 / 中考包 / 整本书包 / 论文包 / 学生包（规划）

### L1 能力层（Capabilities）—— 按工作流环节分域，与现状 4 域对齐但补缺口
- 备课设计域：备课助手、整本书阅读、跨学科设计、配图规划、+课件生成（规划）
- 文言古汉语域（中枢）：文言词典、王力视角、吕叔湘视角、+古诗文鉴赏（规划）
- 评价考情域：江西中考备考、中考分析（通用）、文本词频、+作文批改（规划）
- 教研写作域（含护栏）：教研写作、文稿审核、+学生自学卡（规划）

### L2 数据底座（Foundation）—— 单一数据源，禁止重复
- 文言词库（guwen-cidian references，全库中枢）
- 品牌声音规范（docs/品牌声音与品牌规范.md）
- 通用模板库（教案/论文/报告/审核卡）
- 教材版本库

`<动词>-<对象>` 英文 id + 中文显示名（frontmatter 加 `display-name`）：
- guwen-cidian → 文言词典（保留）

- k12-lesson-planning-zh → 备课助手

### 显式依赖（建议 frontmatter 新增字段）
- `category: 备课设计 | 文言古汉语 | 评价考情 | 教研写作`
- `depends-on: [guwen-cidian]`（doc-review、备课文言相关）
- 这样路由层可自动编排，改名也不破链。
